# PR 1.4.6 — First pixel: note + cents display

**Status:** plan, awaiting user review
**Phase:** 1.4 macOS cabinet
**Predecessor:** 1.4.5 (mic → engine → pitch printed via os_log)
**Successor:** 1.4.7 (full visualiser, four analyzers)

## Goal

Render the live pitch in the SwiftUI window as a **note name + cents** display
(big), with the raw Hz underneath (small). The display updates from a 30 Hz UI
timer reading a lock-free slot the audio thread writes into. School-aware
naming is wired (Western now; Hindustani/Karnatik shaped but not shipped).

## Non-goals

- Hindustani / Karnatik picker UI (shape the abstraction; ship Western only).
- Multi-analyzer visualiser (1.4.7).
- ECS / port-binding components (1.4.7+).
- Visual smoothing, vibrato, onset, breath (later PRs).
- Engine-side school awareness (cabinet concern in 1.4.6).
- Tuning calibration UI (A4 stays 440 Hz; the western context exposes it but
  the picker is deferred).

## Architecture

```
                                     audio thread
   AVAudioEngine tap ──► ring/scratch ──► engine_process_block
                                          │
                                          └─► last pitch Hz
                                              │
                                              ▼
                                       PitchSlot (atomic)
                                              │
                       ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─│─ ─ ─ ─ ─ ─ ─ ─
                                              │
                                              ▼
                                     UI 30 Hz Timer reads slot
                                              │
                                              ▼
                                     PitchNamer.name(hz, ctx)
                                              │
                                              ▼
                                     SwiftUI Text views
```

## Must-fix items from architect review (all accepted)

### 1. Hop-aligned feeding via a scratch buffer

PitchYin's hop is 512; AVAudioEngine delivers ~1024-frame buffers. The current
1.4.5 loop calls `engine_process_block(n)` with `n = min(blockSize, frames -
offset)`, so the *second* call in a callback straddles a hop boundary with
stale internal state. This causes audible/visible stuttering in cents.

**Fix:** maintain a Float scratch buffer sized to `maxTapFrames + blockSize`
(pre-allocated once at `installTap` time, owned by the pipeline, never
re-allocated). Each callback appends converted frames to the scratch, then
drains in `hop`-aligned chunks (`hop = 512`) by feeding the engine `hop`
frames at a time. Leftover (< hop) stays in the scratch for the next callback.

This is a one-cabinet-side fix; the engine itself doesn't change.

### 2. Pitch slot carries `(seq: UInt64, hz: Float)`, packed in a UInt64

A bare atomic `Float` cannot distinguish:
- no new detection this UI tick,
- same pitch held across ticks,
- silence.

**Fix:** pack `seq:UInt32 | hz_bits:UInt32` into a `UInt64` slot. Audio thread
increments seq each time YIN emits a valid Hz (or each block — see below) and
stores the packed value. UI thread reads atomically, unpacks, and compares
seq against its own last-observed seq to know whether to dim/fade.

Open question for review: **seq bumps per detection (only when hz > 0)** vs
**seq bumps per block (with sentinel hz_bits = NaN for "no pitch")**. Leaning
toward per-block with NaN sentinel — UI then knows "silence detected at tick
T" rather than "no information since tick T". Decide before writing code.

### 3. Lock-free via swift-atomics, not OSAllocatedUnfairLock

`os_unfair_lock` can priority-invert; on the render thread that is exactly
the bug the RT-safety rule exists to prevent.

**Fix:** add `swift-atomics` SwiftPM dependency, use `ManagedAtomic<UInt64>`.
(Swift 6 stdlib `Atomic<UInt64>` would also work but requires bumping the
deployment target; staying on macOS 13 keeps the package dep.)

## Should-revise items from architect review (all accepted)

### A. Naming stays in the cabinet, not the FFI

Tonic + school is a UI concern in 1.4.6. The Rust engine emits Hz; the cabinet
converts Hz → note name. Only promote to FFI later if multiple cabinets
(iPad, web) need shared naming logic.

### B. `Register` is a typed enum, not a free-form Int

```swift
enum Register: Equatable, Hashable {
    case western(Int)              // e.g. octave 4 in A4
    case indian(IndianOctave)      // .mandra / .madhya / .taara
}
```

Mixing schemes at the type level is impossible; the namer dispatches on the
context type, not on a generic Int.

### C. Two pitch contexts, not one shared struct with `tonicHz?`

```swift
struct WesternContext {
    var a4Hz: Float = 440.0        // exposed for future calibration UI
}

struct IndianContext {
    var tonicHz: Float             // singer's Sa
    var school: IndianSchool       // .hindustani / .karnatik
}

enum PitchContext {
    case western(WesternContext)
    case indian(IndianContext)
}
```

The single-context-with-optional-tonic shape conflated two genuinely different
naming schemes. Two contexts make the difference visible in the type system.

### D. School picker ships Western-only

A `Picker` in the toolbar with one enabled entry ("Western") and Indian
classical hidden entirely (not greyed out — disabled menu items rot). The
`PitchContext` enum + `PitchNamer` protocol are in place so adding Hindustani
is a small additive PR; no dead UI in 1.4.6.

## Detailed work breakdown

1. **swift-atomics dep.** Add the package via Xcode → Package Dependencies,
   link to the Gurukul target. Commit pbxproj + Package.resolved.

2. **PitchSlot.swift.** `ManagedAtomic<UInt64>` slot, `store(seq:hz:)` and
   `load() -> (seq, hz)`, with `Float` ↔ `UInt32` via `bitPattern`. Sentinel
   `hz_bits = .nan` for "block had no detection". Atomic op = relaxed (single
   producer single consumer, no other ordering deps).

3. **PitchNaming module.** Pure functions, no engine deps.
   - `WesternNamer.name(hz: Float, ctx: WesternContext) -> NamedPitch?`
   - Returns nil for hz <= 0 / NaN; otherwise computes
     `semitonesFromA4 = 12 * log2(hz / ctx.a4Hz)`, rounds to nearest semitone
     to pick the name, residual × 100 = cents.
   - Unit test stub deferred to 1.4.7; sanity check by ear in 1.4.6.

4. **AudioEngine.swift edits.**
   - Pre-allocate `scratch: UnsafeMutableBufferPointer<Float>` at install
     time, sized to `maxTapFrames + hop` (use 8192 to be safe).
   - Track `scratchFill: Int` (count of valid frames).
   - In `handleInputBuffer`, append converted frames into scratch starting at
     `scratchFill`; advance.
   - Drain loop: while `scratchFill >= hop`, copy `hop` frames into the
     engine in-port, call `engine_process_block(hop)`, read the **last sample
     of the pitch out-port**, atomically write `(seq+1, hz)` (or NaN
     sentinel) to the shared `PitchSlot`. Shift leftover frames down.
   - Remove the existing `feedEngine` chunk-by-blockSize logic — it's
     replaced by hop-aligned draining.
   - Drop the per-pitch `log.debug` line; keep it behind a debug-only flag,
     since the UI now shows the pitch. Keep the every-25-buffer tap counter.

5. **ContentView.swift edits.**
   - Replace status-only view with: big `Text` for "C#4 +12¢", small `Text`
     for "277.4 Hz", small `Text` for status / errors.
   - `@State var displayed: NamedPitch?` (nil = "—").
   - `Timer.publish(every: 1/30, on: .main, in: .common).autoconnect()` →
     read `PitchSlot`, unpack, if seq changed run the namer, update state.
   - Track `lastObservedSeq: UInt64` so we know when to fade/dim (e.g. if seq
     hasn't advanced in 200 ms, render as dimmed). Concrete styling can be
     refined visually once it's running.
   - Toolbar: `Picker("School", selection: ...)` with one enabled case
     (Western); IndianContext exists in the type system but no picker entry.

6. **PitchSlot ownership.** Pipeline owns it, exposes a read-only handle to
   the view. SwiftUI doesn't observe the atomic directly — the Timer is the
   bridge. (No `@Published`, no Combine.)

## Optional, only if time allows

- Peak amplitude indicator (we already compute peak every 25 buffers — surface
  it as a tiny meter so users can see "the mic is hearing something").
- Engine-error banner: if `engine_process_block` ever returns non-OK, set a
  visible banner in the UI rather than just logging.
- Start/stop button (currently we start once on `.task` and never stop;
  shouldn't ship a "stuck on" state).
- **Tap-drained gate** before tearing down `enginePtr` in `stop()` /
  `deinit`. Today there's a small race: removeTap → engine_free could
  overlap with an in-flight callback. Add a serial dispatch barrier or an
  atomic "stopping" flag the callback checks before touching `enginePtr`.
  Worth doing even if other optionals slip — it's a real (if rare) crash.

## Realtime safety audit

After implementation, the audio-thread callback must:
- not allocate (scratch buffer is pre-allocated; AVAudioPCMBuffer for
  conversion is the one remaining heap allocation per callback — TODO marker
  stays, fix in 1.4.7 when we move to a pure C-buffer path),
- not take locks (atomics only),
- not call Swift runtime functions that can allocate (no String formatting,
  no os_log on the per-pitch path),
- not call back into UI code.

The `AVAudioPCMBuffer` allocation in 1.4.5 is the one known violation; it
predates this PR and is the only thing blocking strict RT-safety. Leave the
TODO; remove when the converter path is replaced with a pre-allocated
ring + manual resample.

## Acceptance criteria

- Sing a note → window shows e.g. "A4 −7¢" big, "437.2 Hz" small, updates
  smoothly at ~30 Hz with no visible stutter on hop boundaries.
- Silence → display dims / shows "—" within ~200 ms.
- App launches into Listening state on first run (mic permission flow
  unchanged from 1.4.5).
- `cargo fmt --check`, `cargo clippy --workspace -- -D warnings`,
  `cargo test --workspace` all green (no Rust changes expected, but
  re-check).
- Manual smoke: launch app, sing a steady pitch for ~5 s, verify the cents
  reading is stable to within a few cents.
