# PR 1.4.7 — Visualiser (Phase 1.4 closer)

**Status:** plan, architect-reviewed (M1–M4 baked in), awaiting user go-ahead
**Phase:** 1.4 macOS cabinet
**Predecessors:** 1.4.6 (note + cents), 1.4.6.1 (HAL capture), 1.4.6.2 (device-swap)
**Successor:** Phase 1.4 closes; Stage 2 begins.

## Goal

All four analyzers visible in one panel, responding live to the
microphone. The view is one cohesive surface — not four cards — because
these are four aspects of the same voice happening at the same time:

- **Pitch** — scrolling pitch trace over the last ~5 s.
- **Onset** — vertical tick marks layered on the pitch trace at their
  wall-clock x-position.
- **Breath** — thin horizontal bar/strip below the trace that fills
  while breath is detected.
- **Vibrato** — small inset readout: rate (Hz) + depth (cents).

The big-note / cents readout from 1.4.6 stays. Visualiser sits below it.

## Non-goals

- **ECS.** Slipped to 1.5 (see "ROADMAP / PHASE_1_4 edits" below).
- **Recording, replay, export.** No persistence.
- **Per-analyzer enable/disable toggles.** All four are always running
  and always visible.
- **Faceplate / parameter editor UI.** Defaults from `describe-node`
  only. If a default produces obviously-wrong behavior on real voice,
  fix in a 1.4.7.1 follow-up — not here.
- **Engine or schema changes.** This PR is strictly a cabinet change.
  No new node types, no new ports, no schema bumps. Engine API
  (`engine_resolve_out_port`, `engine_out_port`) already exposes
  everything we need.
- **Per-feature confidence / VAD.** The cabinet's RMS gate from 1.4.6
  stays as-is — silence handling is its own piece of work (a real
  voice-activity detector with adaptive noise floor + per-feature
  confidence, queued for Stage 2 — not patching the absolute floor).

## Architect must-fix items (baked into the design below)

These were flagged during plan review and are now load-bearing for the
implementation:

- **M1.** Don't use seqlock. Swift's memory model doesn't sanction
  "plain non-atomic fields gated by an atomic `seq`" — that's UB on
  data race. A correct Swift seqlock requires every field to be a
  `ManagedAtomic<UInt32>` with `bitPattern` loads/stores plus
  fences; it is more code, more failure modes, and harder to review
  than the alternative.
- **M2.** Use a **triple-buffer + atomic latest-index** instead.
  Pre-allocate 3 `FeatureSnapshot` slots and a
  `ManagedAtomic<UInt8>` "latest published index." Writer picks a
  slot that is neither the latest *nor* the one the reader is
  currently claiming, writes its fields plainly into that slot,
  then publishes the slot index with a `.release` store. Reader
  `.acquire`-loads the latest index, then `.acquire`-loads it again
  after claiming the slot to keep the writer from clobbering
  mid-read. Wait-free both sides, no retries, no torn snapshots.
  Details in §"Feature hand-off" below.
- **M3.** World JSON id collision. Per `PHASE_1_4.md:106`:
  *"Boundary port ids are disjoint from node ids."* Initial draft
  had node `onset` *and* boundary port `onset` (same for `breath`).
  Renames in §"World JSON" below: node ids are now `pitch_yin`,
  `onset_det`, `breath_det`, `vibrato_det`. Boundary port ids stay
  the verbs the UI cares about (`pitch`, `onset`, `breath`,
  `vibrato_rate`, `vibrato_depth`).
- **M4.** Onset is event-shaped. The current drain reads
  `pitchBuf[outLen - 1]` (last-sample). Per
  `node-onset/src/lib.rs:1–17`: *"emits a one-sample pulse … 1.0 on
  the sample at which an onset is detected, 0.0 otherwise. Pulses
  are exactly one sample wide."* With hop=512, the chance the pulse
  is at index 511 is ~0.2% — onsets will almost always be missed
  with last-sample reads. Per port, the cabinet must use the read
  shape that matches the node's emit semantics. Pinned table in
  §"Per-port read shape" below.

## Architecture

### World JSON (cabinet-side, in `AudioEngine.swift`)

Extend the embedded world from 1 node / 1 boundary out-port to 4 nodes /
5 boundary out-ports. **Schema unchanged**; we're just authoring a
richer graph against it.

```
in_ports:
  - mic

out_ports:
  - pitch
  - onset
  - breath
  - vibrato_rate
  - vibrato_depth

nodes:
  - pitch_yin    (PitchYin, defaults)
  - onset_det    (Onset,    defaults)
  - breath_det   (Breath,   defaults)
  - vibrato_det  (Vibrato,  defaults)

connections:
  mic               → pitch_yin.audio_in
  mic               → onset_det.audio_in
  mic               → breath_det.audio_in
  pitch_yin.f0      → vibrato_det.f0
  pitch_yin.f0      → pitch
  onset_det.onset   → onset
  breath_det.breath → breath
  vibrato_det.rate  → vibrato_rate
  vibrato_det.depth → vibrato_depth
```

Node ids are renamed (M3) to avoid colliding with boundary port ids.
Boundary port ids are unchanged because the UI side wants the natural
names.

The cabinet resolves 5 out-port handles in `buildEngineIfNeeded()`. The
drain loop reads each per block. **No engine, schema, or FFI change.**

### Per-port read shape (M4)

Verified against node source:

| Boundary port      | Node emit shape                         | Cabinet read |
|--------------------|-----------------------------------------|--------------|
| `pitch`            | sample-and-hold (PitchYin holds f0)     | `buf[hop-1]` (last) |
| `onset`            | **one-sample pulse, 1.0 or 0.0**        | **`max-|x|` over the hop** (any non-zero) |
| `breath`           | sample-and-hold gate (0.0 or 1.0)       | `buf[hop-1]` (last) |
| `vibrato_rate`     | sample-and-hold (`.fill(held_rate)`)    | `buf[hop-1]` (last) |
| `vibrato_depth`    | sample-and-hold (`.fill(held_depth)`)   | `buf[hop-1]` (last) |

For `onset`, "any non-zero in the hop" is the right read. We publish
**the maximum absolute value seen in the hop** so a future spectral-flux
onset variant (which can emit non-binary magnitudes) is handled the
same way. Today the value is 0.0 or 1.0.

### Feature hand-off: triple-buffer

Replace `PitchSlot` (single 64-bit packed atomic, 1.4.6) with
`FeatureSlot`:

```swift
struct FeatureSnapshot {
    var seq: UInt32          // monotonic per publish
    var hz: Float            // pitch
    var onset: Float         // onset magnitude in the last hop (0 or 1 today)
    var breath: Float        // breath gate (0 or 1)
    var vibratoRate: Float   // Hz
    var vibratoDepth: Float  // cents
}

final class FeatureSlot {
    // Three pre-allocated slots, one atomic index.
    private let slots: UnsafeMutablePointer<FeatureSnapshot>   // capacity 3
    private let latest = ManagedAtomic<UInt8>(0)
    // Reader marks the slot it's currently reading so the writer
    // doesn't pick the same one. Single-reader only.
    private let readerClaim = ManagedAtomic<UInt8>(255)

    // Writer (audio thread): pick a slot != latest && != readerClaim,
    // write fields plainly, publish:
    func store(_ snapshot: FeatureSnapshot) {
        let cur = latest.load(ordering: .relaxed)
        let claim = readerClaim.load(ordering: .relaxed)
        let target: UInt8 = (0...2).first { $0 != cur && $0 != claim }!
        slots[Int(target)] = snapshot
        latest.store(target, ordering: .release)
    }

    // Reader (UI thread):
    func load() -> FeatureSnapshot {
        while true {
            let idx = latest.load(ordering: .acquire)
            readerClaim.store(idx, ordering: .relaxed)
            // Re-check: writer may have published a new slot before
            // we set readerClaim. If so, it can't have used `idx`
            // (it picked != latest), so the snapshot at `idx` is
            // stable for us — but we want the freshest available.
            let idx2 = latest.load(ordering: .acquire)
            if idx2 == idx {
                let s = slots[Int(idx)]
                readerClaim.store(255, ordering: .relaxed)
                return s
            }
            // Else loop with the newer index. At most one extra
            // iteration in steady state (1 writer + 1 reader).
        }
    }
}
```

**Why this and not seqlock or 5× independent atomics:**

- 5× independent packed atomics (each like `PitchSlot`) let the reader
  see a torn snapshot (pitch from hop N, vibrato from hop N+1).
  Across-hop tearing is exactly the bug the visualiser would surface
  — onset@N with pitch@N+1 produces ghost ticks at the wrong
  x-position.
- Seqlock in Swift requires `ManagedAtomic<UInt32>` per field with
  bitPattern dance and explicit fences (Swift memory model does not
  permit plain reads/writes racing with atomics). Doable, but more
  code and more places to get the ordering wrong.
- Triple-buffer: writer picks a free slot, writes plain, publishes
  index. Reader reads the published slot. Wait-free both sides;
  zero retries in steady state; one atomic store on publish, one
  atomic load on observe. No data race in the language's model
  because the only memory shared concurrently is the atomic index
  and the slot-not-currently-published.

**RT-safety of the writer:** one `relaxed` load, one `relaxed` load,
one plain struct store (5 floats + 1 UInt32), one `release` store. All
allocation-free.

**Single-reader assumption:** holds today (`ContentView` is the only
reader, on `MainActor`). If we ever add a second reader (e.g. a
recording sink) we move to a different pattern — explicit in the
type's doc.

```
│ ┌─────────────────────────────────────────────────────┐ │
│ │                                                     │ │  ← Pitch trace
│ │     ····───╱╲────                                   │ │     (5 s window)
│ │             ╲                                       │ │     Onset ticks
│ │        |    ───╲___                                 │ │     layered on top
│ │                                                     │ │
│ └─────────────────────────────────────────────────────┘ │
│ [breath ▓▓░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░] │  ← Breath strip

│ Listening — sing or hum into the mic.                   │
└─────────────────────────────────────────────────────────┘
```

- **Pitch trace** — `Canvas` view drawing from a ring buffer of the
  last N pitch samples (`N = 5 s × 30 Hz = 150` points). Y-axis: cents
  off target note (uses the existing `NamedPitch.cents` from
  `PitchNamerFactory`), clipped to ±100¢. **NaN handling:** NaN /
  unvoiced samples produce a gap in the path via `move` (not
  `addLine`) on NaN-following-non-NaN. One `Path`, segmented by
  `move`s.
- **Onset ticks** — recorded into a separate ring buffer of
  `(x_index, magnitude)` per UI tick. Drawn on top of the trace in
  the same Canvas.
- **Breath strip** — a `Rectangle` whose `fillAmount` smooths the
  current breath value via an exponential filter (~50 ms time
  constant). Just a visual gate, not a hard threshold.
- **Vibrato readout** — two `Text` views showing rate (Hz, 1
  decimal) and depth (cents, integer). Verified from node source
  (`node-vibrato/src/lib.rs:259`): `outputs[0].fill(self.held_rate)`
  / `outputs[1].fill(self.held_depth)`. Vibrato emits `held_rate=0`
  and `held_depth=0` on no-vibrato segments. **Sentinel:** show "—"
  when `rate <= 0 || rate.isNaN || depth <= 0 || depth.isNaN`.
- All four views read from the same `FeatureSnapshot` per UI tick
  (existing 30 Hz timer in `ContentView`).

### Discontinuity reset (device-swap interaction)

After a 1.4.6.2 device swap, the audio side calls `scratchFill = 0`
but the UI ring buffers (pitch trace, onset ticks) still hold ~5 s of
stale points from the previous device. Two options:

1. **Sentinel-based clear:** writer publishes one extra "discontinuity"
   `FeatureSnapshot` with `hz = .nan` and a flag bit. Reader notices
   and clears.
2. **Seq-jump clear:** writer bumps `seq` by 1000 across a
   discontinuity; reader notices the jump and clears.

Going with **(1) — explicit discontinuity flag.** Add a sixth field
`discontinuity: Bool` to `FeatureSnapshot`. The swap path
(`swapHALDevice`) and the start-of-fallback path set a one-shot flag
read on the next publish.

### Min window height

Current `minHeight: 320` is too short for the visualiser. Bump to
`560` so the trace has room to breathe.

## Realtime-safety audit (updated)

Audio-thread hot path (HAL IO proc → `appendAndDrain`) gains:

- **5 `engine_out_port` calls per drained hop** (was 1). Each is a
  pointer-dereference + length-read; handles are pre-resolved at build
  time and infallible by construction.
- **1 max-scan over `hop=512` floats** for the onset port (M4). Plain
  `for` loop, no allocation.
- **1 plain struct store + 1 release atomic store** per drained hop
  (triple-buffer publish). No allocation, no fence in user code.

What stays the same:

- Scratch buffer + `halBufferStorage` + `halBufferList` pre-allocated.
- Per-hop RMS energy gate.
- `engine_process_block` per drained hop.
- `engine_in_port` per drained hop.
- `os_log` calls only on rare branches (every 200 callbacks / errors).

Net: still allocation-free, lock-free, wait-free on the audio thread.
The `FeatureSlot.store` (triple-buffer publish) is strictly cheaper
than the previous `PitchSlot.store` would have been if we had widened
it to 5 floats with packed atomics (which we can't anyway in 64 bits).

## Acceptance criteria

- App launches; HAL path engages (or fallback); visualiser panel
  renders below the big-note readout.
- **Pitch trace:** sustained sung note shows a stable horizontal line
  near the target cents; phrase with discrete notes shows step
  transitions. NaN gaps render as breaks, not zero-baseline dips.
- **Onset hit-rate:** **10 sung note-attacks in a row produce ≥9
  visible ticks**, each within ~100 ms of the actual attack. (Manual
  test — pick a phrase like a five-note scale ascending, sing it
  twice.)
- **Breath:** audible breath between sung phrases fills the breath
  strip; no breath during sustained sung notes.
- **Vibrato:** add deliberate vibrato on a held note → readout shows
  rate in 4–7 Hz range and a non-zero depth. No vibrato → "—".
- **Silence behavior preserved:** mic muted → big-note view dims to
  "—" (existing 1.4.6 behavior unchanged); pitch trace shows gaps.
- **Device-swap (1.4.6.2):** changing default input mid-session
  clears the pitch trace + onset ticks; new device's audio populates
  a fresh trace within ~1 s.
- **Coherence smoke test (debug-only):** add a `#if DEBUG` counter on
  the reader path: count `latest.load` retries from the triple-buffer
  (the inner `if idx2 == idx { ... }` failure path). Log once on
  `stop()`. In steady state on a 30 Hz reader vs ~94 Hz writer the
  retry count should be <1% of reads.
- `cargo fmt --check`, `cargo clippy --workspace -- -D warnings`,
  `cargo test --workspace` clean (Rust unchanged; verify nonetheless).
- `xcodebuild build` clean.
- `docs/ROADMAP.md` and `docs/PHASE_1_4.md` updated as above.

## Implementation outline

1. **`FeatureSnapshot` + `FeatureSlot`** — new file. Triple-buffer
   pattern as in §"Feature hand-off". Include `discontinuity: Bool`
   field.
2. **World JSON in `AudioEngine.swift`** — extend to 4 nodes / 5
   boundary out-ports per §"World JSON". Rename triggers a fresh
   build of the embedded string; CLI verification via
   `cargo run -p cli -- validate` is not currently part of cabinet
   build, but we should manually paste the JSON through that to
   confirm before committing.
3. **Handle resolution** — `buildEngineIfNeeded` resolves 5
   `pitchHandle`-style handles (rename to descriptive names:
   `pitchOut`, `onsetOut`, `breathOut`, `vibratoRateOut`,
   `vibratoDepthOut`).
4. **`appendAndDrain` drain loop** — read 5 ports per hop, apply the
   read-shape table (M4: max-scan for onset, last for the others).
   Build a `FeatureSnapshot`, publish to `FeatureSlot`.
5. **`swapHALDevice` + start-of-fallback** — set the
   `discontinuity` flag for the next publish.
6. **`ContentView`** — switch from `pitchSlot.load()` to
   `featureSlot.load()`; pass the snapshot into the new visualiser
   subviews; bump `minHeight` to 560.
7. **`Visualiser/PitchTraceView.swift`** — Canvas + ring buffer +
   NaN-gap drawing + onset-tick overlay. Clear ring on
   `discontinuity`.
8. **`Visualiser/BreathStripView.swift`** — exponential-smoothed
   Rectangle fill.
9. **`Visualiser/VibratoReadoutView.swift`** — two Text views with
   the "—" sentinel.
10. **ROADMAP.md + PHASE_1_4.md §7 edits** — same commit.
11. **Manual test pass against acceptance criteria** before
    committing.
