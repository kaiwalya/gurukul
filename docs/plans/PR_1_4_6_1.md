# PR 1.4.6.1 — CoreAudio HAL mic capture

**Status:** plan, architect-reviewed (M1/M2/M3 baked in), awaiting user go-ahead
**Phase:** 1.4 macOS cabinet
**Predecessor:** 1.4.6 (note + cents display, AVAudioEngine tap path)
**Successor:** 1.4.7 (full visualiser)

## Goal

Replace `AVAudioEngine.installTap(onBus:)` with a CoreAudio HAL output
AudioUnit (`kAudioUnitSubType_HALOutput`, input-enabled) inside
`AudioPipeline`. The tap path queues ~200 ms of audio on the default
display-mic before delivering buffers; the HAL unit delivers ~10 ms IO
callbacks straight from the device, eliminating the dominant remaining
source of perceived input-to-display lag.

This is a fix carried over from PR 1.4.6's commit body — a follow-up, not
a new milestone. 1.4.7 (visualiser) stays in its slot.

## Non-goals

- Device-picker UI. Use the system default input; log which device we got.
- A `MicSource` protocol / abstraction. One implementation is not enough
  evidence to design against; the abstraction comes when iOS or another
  cabinet gives us the second use case.
- iOS / Android cabinets. Out of scope.
- Stereo / non-48 kHz device support. If the default device isn't mono
  Float32 48 kHz we fall back to AVAudioEngine (kept as a one-shot
  fallback inside `AudioPipeline`, not a separate path the user picks).
- Engine-side energy gate (still a queued follow-up).
- **Default-device-change listener.** If the user changes default input
  mid-session (AirPods connect, USB mic plug, System Settings change),
  the HAL unit keeps reading the device it was given at start. This is
  a known, named gap — wiring
  `kAudioHardwarePropertyDefaultInputDevice` as a property listener is
  a follow-up. Today's behavior: log the device once at start, never
  re-evaluate.
- Tap-drained teardown gate as a standalone follow-up. **This PR
  subsumes it for the HAL path** via M3 below; the AVAudioEngine
  fallback path inherits the existing weaker behavior.

## Why HAL and not other options

- **`AVAudioEngine.installTap`** is what 1.4.6 uses. Designed for graph
  taps, not low-latency input; system queues ~200 ms before delivering on
  the LG display mic. `bufferSize` is treated as a hint and ignored here.
- **`AVAudioEngine.inputNode` rendered to file / processed via render
  block** — same underlying queueing, just different consumption.
- **`AVAudioSourceNode` / pulling from `AVAudioEngine`** — pull model
  doesn't expose a lower-latency device read.
- **`AudioUnit kAudioUnitSubType_HALOutput`** — direct CoreAudio device
  access. The IO proc fires at the device's hardware buffer size
  (`kAudioDevicePropertyBufferFrameSize`, observed 512 frames = 10.7 ms on
  the LG mic). This is the right primitive. macOS-only; iOS would use
  `kAudioUnitSubType_RemoteIO` later.
- **`cpal` / `miniaudio` / `JUCE`** — out of scope per
  `PHASE_1_4.md` ("the engine never links CoreAudio, cpal, or any
  platform audio API"; the macOS cabinet owns its audio). HAL is a
  cabinet-internal choice and stays in `apps/mac/Gurukul/`.

## Architecture

```
   Default input device (system pref or LG mic)
       │
       ▼
   AudioUnit (kAudioUnitSubType_HALOutput, input-enabled)
       │  IO proc: ~10ms buffers, native format (Float32, mono, 48 kHz)
       ▼
   AudioPipeline.handleHALBuffer(...)
       │  unchanged downstream: scratch → hop-aligned drain →
       │  energy gate → PitchSlot
       ▼
   PitchSlot ─────► UI (unchanged)
```

`AudioPipeline` keeps its lifecycle methods (`start`, `stop`), its
scratch buffer, its hop-aligned drain, the per-hop RMS energy gate, and
the `PitchSlot`. The only thing that changes is **how frames arrive**.

## Architect must-fix items (baked in below)

These were flagged during plan review and are now load-bearing for the
implementation:

- **M1.** AudioUnit property scope/element pinning. The "set stream
  format on bus 1's output scope" wording is right but the most
  common HAL-setup bug. Pin the table in §1 below so the reader does
  not have to re-derive it.
- **M2.** `AudioBufferList` sizing. `inNumberFrames` on the IO proc is
  not pinned to the value of `kAudioDevicePropertyBufferFrameSize` at
  start — it can change on route changes, sample-rate changes, audio
  server restarts, etc. Set
  `kAudioUnitProperty_MaximumFramesPerSlice` on the unit explicitly
  and size the `AudioBufferList` to that, not the current buffer
  frame size. Defend the IO proc against
  `inNumberFrames > maxFramesPerSlice` (shouldn't happen, but bail
  and log if it does — don't render and silently truncate).
- **M3.** Tear-down race. `AudioOutputUnitStop` mostly drains the IO
  proc but documented edge cases (route change, aggregate-device
  reconfig) let a final IO proc fire after Stop returns. With
  `Unmanaged.passUnretained(self)`, a late callback that lands after
  the unit is disposed and the pipeline deallocates is a
  use-after-free. Add a `ManagedAtomic<Bool> stopping` flag; IO proc
  reads it relaxed at entry and bails immediately if set; `stop()`
  sets the flag *before* calling `AudioOutputUnitStop`. Subsumes the
  tap-drained-gate optional from 1.4.6 for the HAL path.

## Implementation outline

1. **HAL unit construction in `AudioPipeline`.**
   - `AudioComponentDescription { type: kAudioUnitType_Output, subType:
     kAudioUnitSubType_HALOutput, manufacturer: kAudioUnitManufacturer_Apple }`
     → `AudioComponentFindNext` → `AudioComponentInstanceNew`.
   - **M1 — property table:** apply in this exact order before
     `AudioUnitInitialize`. Spelling out scope + element to prevent
     the most common HAL-setup bug:

     | Property                                  | Scope          | Element | Value                |
     |-------------------------------------------|----------------|---------|----------------------|
     | `kAudioOutputUnitProperty_EnableIO`       | Input          | 1       | `1` (enable input)   |
     | `kAudioOutputUnitProperty_EnableIO`       | Output         | 0       | `0` (disable output) |
     | `kAudioOutputUnitProperty_CurrentDevice`  | Global         | 0       | `<deviceID>`         |
     | `kAudioUnitProperty_StreamFormat`         | Output         | 1       | `<clientASBD>`       |
     | `kAudioUnitProperty_MaximumFramesPerSlice`| Global         | 0       | `maxFramesPerSlice`  |
     | `kAudioOutputUnitProperty_SetInputCallback`| Global        | 0       | `<cb + refcon>`      |

     The non-obvious one is `StreamFormat`: bus 1 is *input from the
     device*, so element 1's **Output** scope is what the unit
     produces to *our* callback — that's the side we set, not Input.
   - `<deviceID>` comes from
     `kAudioHardwarePropertyDefaultInputDevice` at start time. Read
     and capture once; we do not re-evaluate (see Non-goals).
   - `<clientASBD>` is mono Float32 non-interleaved at the engine's
     `sampleRate` (48 kHz). If the device's native format differs the
     unit performs conversion internally — no AVAudioConverter needed
     in the cabinet.
   - `maxFramesPerSlice` is a generous bound (4096) — see M2.
   - If `AudioUnitInitialize` fails for *any* reason (format
     rejected, device gone, exotic aggregate device), log loudly and
     fall back to AVAudioEngine.

2. **Pre-allocations done in `start()` (before `AudioOutputUnitStart`).**
   - **M2 — sizing:** size the backing storage for the
     `AudioBufferList` to `maxFramesPerSlice` frames (4096), not the
     observed `BufferFrameSize`. `inNumberFrames` on the IO proc can
     mutate at runtime (route change, SR change). With
     `MaximumFramesPerSlice` set explicitly, the unit will not deliver
     more than that; the IO proc still defends against the
     hypothetical-but-shouldn't-happen overflow by checking
     `inNumberFrames > maxFramesPerSlice` at entry and bailing.
   - `AudioBufferList` with one `AudioBuffer`: `mNumberChannels = 1`,
     `mDataByteSize = maxFramesPerSlice * sizeof(Float)`,
     `mData = <pre-allocated Float storage>`.
   - Scratch buffer sized for `maxFramesPerSlice + hop` (4608 frames).
     Same role as 1.4.6's scratch; only the upper bound changes.
   - `stopping: ManagedAtomic<Bool>` initialized to `false`.

3. **Threading.**
   - HAL IO procs run on a CoreAudio realtime thread, not on
     `MainActor`. `AudioPipeline` is already declared `nonisolated`
     from 1.4.6, so no isolation churn.
   - The IO proc must not allocate, lock, or call Swift-runtime
     functions that can. Same rules as the existing tap callback —
     hand-written `update`/`UnsafeMutableBufferPointer` operations and
     the `PitchSlot.store` atomic only.
   - The callback is registered via a C function pointer plus a
     `void *` refcon
     (`Unmanaged<AudioPipeline>.passUnretained(self).toOpaque()`),
     standard CoreAudio pattern.

   **Construction order is load-bearing:**
   `buildEngineIfNeeded()` (sets `enginePtr`, port handles) →
   `installHALInput()` (allocates scratch, buffer list, configures
   unit, registers callback) → `AudioOutputUnitStart` (IO proc starts
   firing). The IO proc reads `enginePtr` and `scratch` on the very
   first call; both must already be in place.

4. **IO proc body (the trampoline + the member it calls).**

   ```
   // C entry point. Pulls AudioPipeline out of refcon and dispatches.
   private static let ioCallback: AURenderCallback = { refcon, flags, ts, busNum, nFrames, _ in
       let pipeline = Unmanaged<AudioPipeline>.fromOpaque(refcon).takeUnretainedValue()
       return pipeline.handleHALInput(flags: flags, ts: ts, busNum: busNum, nFrames: nFrames)
   }
   ```

   Inside `handleHALInput`:
   - **M3:** if `stopping.load(.relaxed)` is true, return `noErr`
     immediately. Don't render, don't touch pipeline state.
   - If `nFrames > maxFramesPerSlice` (defend against M2 edge case),
     log once and return `noErr`.
   - Reset the `AudioBufferList`'s `mDataByteSize` to `nFrames *
     sizeof(Float)` (the unit expects the buffer list's
     `mDataByteSize` to describe the requested write size on entry).
   - Call `AudioUnitRender(unit, flags, ts, busNum, nFrames,
     &bufferList)`.
   - On non-zero OSStatus, log and bail.
   - Hand the buffer's `mData` (typed as
     `UnsafeMutablePointer<Float>`) to the existing
     `appendAndDrain(src:count:)` — unchanged from 1.4.6.

5. **Lifecycle.**
   - `start()`:
     1. `buildEngineIfNeeded()` (1.4.6, unchanged).
     2. `installHALInput()` → returns `true` if HAL is wired, `false`
        if we should fall back.
     3. If HAL: `AudioUnitInitialize` then `AudioOutputUnitStart`.
        Log `"HAL engaged: device='X' native=<ASBD> client=<ASBD>
        internalConversion=<bool>"` (S2 below).
     4. Else: existing `installTap()` + `avEngine.start()`, log
        `"HAL setup failed: <reason>; falling back to AVAudioEngine
        tap (HIGH LATENCY)"`.
   - `stop()` for HAL path (M3 order matters):
     1. `stopping.store(true, .relaxed)`.
     2. `AudioOutputUnitStop` (waits for in-flight IO procs; any that
        slip through past Stop bail at M3's first-line check).
     3. `AudioUnitUninitialize`.
     4. `AudioComponentInstanceDispose`.
     5. `engine_reset`.
   - `stop()` for fallback path: existing 1.4.6 sequence.
   - `deinit` frees scratch + buffer-list backing storage; the
     unit/component is already disposed by `stop()`.

6. **Fallback path.**
   - If any HAL setup step fails, the fallback is exactly the 1.4.6
     `installTap()` path (including its AVAudioConverter logic,
     conversion-bypass branch, and per-25-callback diagnostics).
   - The fallback log line includes which step failed
     (`AudioUnitInitialize / SetProperty(StreamFormat) / …`) so we
     can debug after the fact.
   - **S2 — diagnostic logging (HAL path):**

     ```
     "HAL engaged: device='LG UltraFine Display Audio' id=72
        native=<48000 Hz, 1 ch, Float32, non-interleaved>
        client=<48000 Hz, 1 ch, Float32, non-interleaved>
        internalConversion=false maxFramesPerSlice=4096"
     ```

     Logs both native and client ASBD side-by-side so the
     `internalConversion=true` case (e.g. a 44.1 kHz built-in mic) is
     visible without needing extra instrumentation.

7. **Log throttling (O1).** With HAL delivering at ~10 ms intervals,
   the existing every-25-callback peak + `scratchFill` log fires
   ~4×/sec, up from ~0.5×/sec under AVAudioEngine. Bump to
   every-200-callback (~2 sec apart) for the HAL path. Fallback path
   keeps every-25.

## File layout

Cabinet-only change:

```
apps/mac/Gurukul/
  AudioEngine.swift            (heavy edit: HAL setup + IO proc + fallback)
```

No new files. No Rust changes. No FFI changes. No new SwiftPM deps —
CoreAudio is part of the macOS system frameworks (`CoreAudio.framework`,
`AudioUnit.framework`, `AudioToolbox.framework`). These need to be linked
via pbxproj edits.

If `AudioEngine.swift` starts to feel crowded (~500 lines after this
change), we can split out a small file later — but per project rules
(no premature abstraction), keep it one file until there's a second
reader.

## Detailed work breakdown

1. **pbxproj: link the three system frameworks.** `AudioToolbox`,
   `CoreAudio`, `AudioUnit`. Same pattern as the existing
   `swift-atomics` link from 1.4.6 (PBXBuildFile entries +
   PBXFrameworksBuildPhase).

2. **AudioEngine.swift: add HAL setup.**
   - New private method `installHALInput() throws -> Bool` that returns
     `true` if HAL was wired up, `false` if we should fall back. (Throws
     only on truly unrecoverable conditions; format/device mismatches
     return `false`.)
   - Pre-allocate `halBufferList: AudioBufferList` (or its
     `UnsafeMutableAudioBufferListPointer` equivalent) plus its backing
     storage at install time. Owned by the pipeline; freed in `deinit`.
   - C callback at file scope (or as a `static` method) that pulls
     `AudioPipeline` from the refcon and calls a member method.

3. **AudioEngine.swift: rewire `start()`.**
   - Try `installHALInput()` first.
   - If it returns `false` or throws on a non-fatal step, fall back to
     the existing `installTap` path.
   - Log which path is active (`"HAL input engaged"` /
     `"AVAudioEngine tap fallback"`).

4. **AudioEngine.swift: rewire `stop()`.**
   - If HAL was active: `AudioOutputUnitStop`, `AudioUnitUninitialize`,
     `AudioComponentInstanceDispose`, then `engine_reset`.
   - If tap fallback was active: existing path.

5. **Drop AVAudioConverter wiring when HAL is active.** Since the unit
   negotiates the stream format up-front, the cabinet always gets mono
   Float32 at the engine SR. The bypass branch from 1.4.6 stays as
   protection for the fallback path.

6. **Diagnostic logging.**
   - Log the device id + name from
     `kAudioObjectPropertyName` once at start.
   - Log the device's `BufferFrameSize` once at start.
   - Keep the existing every-25-callback peak + `scratchFill` log.
   - Add a one-shot log of the actual measured tap-to-tap interval after
     a few hundred callbacks, so we can confirm in the log that the new
     path is delivering ~10 ms not ~200 ms.

## Realtime safety

The HAL IO proc has the same constraints as the existing tap callback:

- No allocations (scratch + AudioBufferList pre-allocated; PitchSlot is
  a single atomic store).
- No locks.
- No Swift-runtime calls that can allocate. The IO proc is a `@convention(c)`
  trampoline that hands off to a pipeline method — that method is the
  same hot path the tap callback uses today, audited in 1.4.6.
- `AudioUnitRender` itself is RT-safe and is the intended way to pull
  input from a HAL unit's IO proc.

Net change: same realtime audit as 1.4.6, with `AudioUnitRender`
replacing the AVAudioEngine tap callback boundary. AVAudioConverter (the
one known RT-unsafety in 1.4.6) is no longer on the hot path when HAL
is active, so this PR also closes that audit item for the default path.

## Acceptance criteria

- App launches, logs `"HAL input engaged"`, shows live pitch with
  perceived sub-100 ms input-to-display lag (down from ~250–350 ms in
  1.4.6).
- App also still works if HAL setup fails — falls back to AVAudioEngine
  tap and logs the reason.
- Mic muted → display dims to "—" within ~700 ms (no regression from
  1.4.6's energy gate).
- Tap-spacing log shows ~10 ms intervals (was ~200 ms in 1.4.6).
- `cargo fmt --check` / `cargo clippy --workspace -- -D warnings` /
  `cargo test --workspace` all green (Rust unchanged; check
  nonetheless).
- `xcodebuild build` clean.

## Resolved decisions (from architect review)

1. **Fallback policy.** Keep AVAudioEngine path as a runtime fallback.
   Failures route there and log the reason. Keeps the app usable on
   exotic devices and gives us an A/B harness for free.
2. **Device-pick on startup.** Use the system default input. Device
   override is not in scope for this PR.
3. **File layout.** One file (`AudioEngine.swift`). No `HALInput.swift`
   sibling. Split when a second reader exists.
4. **Conversion visibility.** Log both native and client ASBD at start
   (see S2 in §6). No warning — `internalConversion=true` is
   informational, not an error.
